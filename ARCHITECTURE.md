## What this site is

A static HTML site with no build step, no framework, no server-side code. Every page is a plain `.html` file. Shared behaviour lives in `nav.js`, `site.css`, and `site-utils.js` at the repo root. Hosted on GitHub Pages.

## nav.js ‚Äî how root detection works

`nav.js` is loaded as a plain (synchronous) `<script>` tag at the bottom of every page. The key line:

```js
const NAV_ROOT = new URL('.', document.currentScript.src).href;
```

`document.currentScript` is only valid during initial script execution, so it's captured immediately at the top level ‚Äî not inside a callback. This gives the URL of the folder containing `nav.js`, which is always the repo root, regardless of how deep the calling page is.

Every link in the nav is then prefixed with `root`, so `root + 'index.html'` resolves correctly whether the page calling it is at `/notes.html` or `/portabletemple/deepdives/forbidden-city.html`.

**Important:** `nav.js` must not have `defer` or `async` on its `<script>` tag, as `currentScript` is null in those contexts.

The nav injects a `<footer>` into `#site-footer` on `DOMContentLoaded`. Pages that also have a `#site-header` element will get the nav injected there too (currently unused).

## Welcome overlay ‚Äî index.html

The overlay is a full-screen modal that fires on first visit to `index.html`. It has three panels: a left sidebar nav, a centre content pane (which loads sub-pages via `fetch`), and a right hero image panel.

### Show/hide logic

```js
// On load:
if (window.location.hash === '#welcome') {
    showOverlay();           // secret/test door ‚Äî ignores snooze
} else if (!window.location.hash && Date.now() > snoozedUntil) {
    showOverlay();
}
```

- **Any hash other than `#welcome`** suppresses the overlay. This means deep-linked pages (e.g. `index.html#about`) land directly on the page content without interruption.
- **`#welcome`** is a backdoor for testing ‚Äî always shows the overlay regardless of snooze state.
- **No hash + not snoozed** ‚Üí overlay shows normally.

### Snooze (24-hour suppress)

The "Don't show this again today" checkbox writes a timestamp to `localStorage`:

```js
localStorage.setItem('overlaySnoozedUntil', String(Date.now() + 24*60*60*1000));
```

On next load, `snoozedUntil` is read and compared to `Date.now()`. If still in the future, the overlay is skipped.

### Content panel loading

Sidebar links use `data-` attributes to declare their behaviour:

| Attribute | Behaviour |
|---|---|
| `data-panel="key"` | Fetch the page, extract `.content` (or `data-selector`), render in centre pane. Cached in `panelCache` after first load. |
| `data-nav="true"` | Navigate away directly (no panel loading). |
| `data-dismiss="true"` | Close the overlay, revealing the page beneath. |
| `data-iframe="true"` | Embed the page in an `<iframe>` in the centre pane. |

The `panelCache` object is seeded with the default home content so re-clicking Home doesn't re-fetch.

## pages-manifest.json ‚Äî auto-generated index

The GitHub Action `.github/workflows/generate-manifest.yml` runs on every push to `main`. It:

1. Walks all `.html` files in the repo (excluding `404.html` and hidden/vendor folders).
2. Extracts the `<title>` tag from each file.
3. Gets the last git-modified date via `git log -1 --format=%ci -- <path>`.
4. Infers a "track" (books/notebooks/general) from the file path.
5. Writes `pages-manifest.json` and commits it back with `[skip ci]` in the message to avoid an infinite loop.

The bot commit identity is `github-actions[bot] <github-actions[bot]@users.noreply.github.com>`.

`pages-manifest.json` is the source of truth for `sitemap.html` and for any future `sitemap.xml` generation.

## Images

**Rule: `assets/` is for site-wide assets only.** Anything tied to a specific research track or tour belongs inside that track's own folder, not in `assets/`. This keeps each project self-contained as more tracks and image sets are added.

`assets/images/` holds only genuinely site-wide images:

- `hero-image.jpg`, `hero-image2.jpg` ‚Äî hero / overlay images used on `index.html`
- `explore.png` ‚Äî UI icon

`portabletemple/images/` holds everything specific to the Portable Temple research track, organised per sub-project:

- `bird-feather/` ‚Äî motif images for the Birds Without Borders tour (Garuda, Nike, Seraphim, Apkallu)
- `forbidden_city_gates/` ‚Äî gate photographs referenced by `portabletemple/data/learn-more.json`
- `landing-giga.png`, `landing-giga-lo-res.jpg` ‚Äî large base map image used by the bird-feather tour

Future research tracks should follow the same pattern: a `<track>/images/<sub-project>/` folder, never dumped into `assets/`.

The Leaflet marker icons live separately in `assets/leaflet/images/` and are referenced by relative paths inside `leaflet.css` ‚Äî do not move them.

**Image path rule for `learn-more.json`:** paths are resolved relative to the HTML page that loads the JSON (`portabletemple/timeline-pt.html`), not relative to the JSON file itself. A path like `images/forbidden_city_gates/file.jpg` resolves correctly from the timeline page; `../images/...` does not.

## Image standardization ‚Äî lm_ingest.py

`tools/lm_ingest.py` is the single tool for turning raw images into web-ready assets. All images published to the site should pass through it.

**What it always does (every mode):**
- Resizes so the long edge is ‚â§ 1600 px (downscale only).
- Bakes in EXIF orientation, then strips all metadata including GPS.
- Sanitizes filenames to lowercase with hyphens ‚Äî no pre-naming required.
- PNG stays PNG; everything else becomes JPEG at quality 85, progressive.

**Four modes ‚Äî same `--src`, swap the destination flag:**

```bash
# A) Standardize only ‚Äî cleaned images into a folder, nothing else
python3 tools/lm_ingest.py --src "<in>" --out "<folder>"

# B) Slideshow tour ‚Äî images + captions, no map
python3 tools/lm_ingest.py --src "<in>" --tour "<folder>/tour.html"

# C) Map tour ‚Äî geographic Leaflet tour; seed coordinates to be corrected per stop
python3 tools/lm_ingest.py --src "<in>" --map-tour "<folder>/maptour.html"

# D) Learn More entry ‚Äî writes learn-more.NEW.json beside the master; never overwrites master
python3 tools/lm_ingest.py --src "<in>" --site <site-id> --project "<...>/portabletemple"
```

Optional tuning flags work in every mode: `--max 2000`, `--quality 90`, `--dry-run`.

**Finishing a generated tour:**
- Slideshow (`--tour`): edit the `slides` list in the generated HTML ‚Äî replace `"TODO"` captions, reorder entries. Image paths are already correct.
- Map tour (`--map-tour`): edit the `stops` list ‚Äî set real `lat/lng/zoom`, replace `"TODO"`. Seed coordinates are placeholders (recycled from three defaults: Chang'an / Heian-ky≈ç / Forbidden City) and will be wrong; correct them per stop.

**Mode D safety:** `learn-more.NEW.json` is always a new file. Review it, then paste the block into the master `learn-more.json` by hand. The script never touches the master.

## Data loading ‚Äî CSV, JSON, and the Google handoff

Both `timeline-pt.html` and `notes.html` load from multiple sources simultaneously via `Promise.allSettled` and a `fetchCSVRows` helper that returns `[]` on any missing or failed file. A partial migration never breaks a page.

**Timeline merges:** Google Sheets URL + `portabletemple/data/sites.csv` + `portabletemple/data/themes.csv`.

**Notes merges:** live Google Sheet + `notes-quick.csv` + `notes-data.json`.

To retire a Google source: export the tab as CSV into `data/`, and the existing loader picks it up. Delete the rows from Google afterward ‚Äî both sources load at once, so leaving them in Google doubles the entries.

**CSV encoding rule:** files are UTF-8 without BOM. Viewing in Excel is safe; saving in Excel is not unless you explicitly choose *CSV UTF-8 (Comma delimited)*. Edit CSVs in a plain-text editor or be deliberate about the save format. Greek, Chinese, Arabic, and punctuation will silently corrupt otherwise.

**RFC-4180 parser:** both pages use a character-by-character RFC-4180 CSV parser (not line-splitting). This correctly handles quoted fields that contain commas, newlines, or paragraph breaks ‚Äî which the Forbidden City deep-dive entry requires. Do not revert to line-splitting.

**`white-space: pre-line`** is set on both `#panel-body` (desktop) and `#mobile-detail-body` (mobile) so newlines inside quoted CSV cells render as paragraph breaks rather than one clump.

## Standalone tours (`tour_*.html`) with Deep Dive / Learn More

**Always test these by running a local server, never by double-clicking the file or dragging it into a browser tab.** If the address bar doesn't start with `http://` or `https://`, Learn More (and anything else that does a `fetch()` of a local file, like `data/learn-more.json`) will silently fail and just never appear ‚Äî no error, no clue, looks exactly like a missing button. Deep Dive will still work fine in that situation, which is the tell: if Deep Dive works but Learn More doesn't, this is almost certainly why. From the repo root:
```bash
python3 -m http.server 8000
```
then open `http://localhost:8000/portabletemple/tour_serpent_forms.html` (swap in whichever tour). This was confirmed as the actual cause of a "Learn More button doesn't exist" report on June 21, 2026 ‚Äî it wasn't missing, it just couldn't read its data file under `file:///`. Once the site is live on GitHub Pages this stops mattering, since that's always served over HTTP ‚Äî but it will bite again during local testing on any future tour with the same pattern.

A `tour_*.html` file (e.g. `tour_prometheus_eras.html`, `tour_serpent_forms.html`) is a fully self-contained Leaflet page, same family as the original `tour_birds-without-borders.html` template. It is never loaded inside `timeline-pt.html`, so it can't reuse that page's overlay elements ‚Äî any Deep Dive or Learn More UI has to be ported directly into the tour file itself. The patterns below were arrived at over several rounds of correction on the Prometheus tours; follow them rather than re-deriving from scratch on the next tour.

**Deep Dive opens as an iframe-in-a-panel, never a new tab.** A standalone tour's deep dive is a local static page, not a Google Doc, so the right model to copy is the site's `story_url` iframe pattern ‚Äî not `timeline-pt.html`'s `deepdive_url` fetch-and-strip-Google-Docs-HTML pattern, which doesn't apply here. (First got this wrong by using `window.open()`; see "Open investigations" below.)

**A deep dive is a dead-end spoke. The tour is the hub.** A deep dive page should end with one plain instruction ‚Äî "Click anywhere outside this panel to return to the tour and keep exploring" ‚Äî and nothing else. Specifically, do not add:
- cross-links from one deep dive to another ("Continue exploring ‚Üí other topic"). Confusing, and breaks the hub-and-spoke model ‚Äî a spoke only ever points back to the hub, never sideways to another spoke. (If deep dives are ever consolidated into fewer, longer pages, that's a deliberate future redesign, not something to back into via cross-links.)
- a "Back to the tour" link inside the deep dive page's own markup, or an "Open full page" link in the overlay header. Both look like navigation but aren't: clicking either *while the page is sitting inside the iframe* just loads a fresh copy of the tour into that narrow side panel, with no tour state ‚Äî it does not return the visitor to the stop they were on. The only correct way back is closing the overlay (click the dark backdrop outside the panel, or the ‚úï). If a deep dive page might ever be opened fully on its own, outside any tour overlay, rely on the browser's own Back button for that case ‚Äî a same-markup link can't tell which context it's in, and a wrong link is worse than no link.

**Deep Dive / Learn More: a non-clickable title pill, then a separate small clickable button ‚Äî never one button carrying the whole title.** This directly copies `timeline-pt.html`'s "Axis Mundi" + "Read more" pattern under "Shared Themes": the pill names the thing, the button (smaller, fixed text, auto-width ‚Äî never `width:100%`) is the only clickable part. Give each motif with a `deepdiveUrl` a short `deepdiveTitle` too (e.g. "Prometheus vs. Satan", "Hesiod, Aeschylus, Shelley" ‚Äî shortened from that deep dive's own longer `<h1>`, which can stay more descriptive on the page itself); render it as `<span class="action-label">{deepdiveTitle}</span><button class="action-btn">üìñ Deep Dive ‚Üí</button>`. Learn More's label just reuses the current stop's own `label` (it's always "more about this same stop," never a different topic the way a Deep Dive link can be). A generic "üìñ Deep Dive ‚Üí" with no title implies every deep dive is interchangeable ‚Äî don't fold the title into the button text either; that produces one long button stretching the panel width, which is exactly the look this pattern exists to avoid.

**`#panel-actions` (the Deep Dive / Learn More row) must NOT be nested inside `#panel-scroll`.** First build put it right after the body text, inside the scrollable region ‚Äî which looks fine on a short stop, but on a longer one (especially in guided-tour mode, where the fixed forks row below eats into the available height too) it gets pushed below the fold inside an inner scrollbar nobody thinks to scroll. It's easy to miss entirely and looks like the button doesn't exist. Keep `#panel-actions` as its own sibling between `#panel-scroll` and `#tour-forks` ‚Äî scrollable body text above it, always-visible actions and forks below it, never competing for the same scroll space. Also hide the whole block (border included) when a stop has neither a Deep Dive nor a Learn More, rather than leaving an empty divider line.

**Set `overflow-x: hidden` on `#panel-scroll`, always.** Setting `overflow-y: auto` without also setting `overflow-x` lets the browser quietly compute `overflow-x` as `auto` too (a real CSS Overflow spec rule, not a bug in one browser) ‚Äî so the smallest horizontal overflow (a fraction of a pixel of rounding is enough) draws an unwanted horizontal scrollbar across the bottom of the scroll box. This is the "mystery white bar" that showed up more than once across these tours; it isn't a progress bar, isn't load-bearing, and the fix is just declaring `overflow-x: hidden` explicitly so the browser has no ambiguous axis to guess about.

**Why the `file:///` issue happens, technically:** `fetch()` of a local JSON file is blocked by the browser when the page itself was opened via `file:///...`, for security reasons. The code's own `.catch` swallows that error quietly (by design ‚Äî so a genuinely missing file doesn't break the page), which is exactly why it produces no visible error anywhere. `openDeepDive()` never hits this because it only opens a URL, it doesn't read a local file. See the note at the top of this section for the always-do-this-first test.

**Tour navigation: Back and the forward option(s) sit in one horizontal row, never stacked.** Back is a real pill button, same visual weight as the forward option(s) ‚Äî not a small muted text link ‚Äî and stays in place, toggling visible/invisible rather than disappearing and shifting the layout. A stop with a single "Next" renders one button beside Back; a stop with real forks renders 2-3 buttons in that same row, wrapping to a second line only if they don't fit ‚Äî never a deliberate vertical stack.

**The stop counter is a marker's fixed position in that tour's own array, full stop.** "3 / 7" means "marker #3 of 7 total markers defined in this tour file" ‚Äî `motifs.indexOf(currentStop) + 1` over `motifs.length`. Both numbers auto-update if a stop is added or removed from the array; no hand-maintained total anywhere. It deliberately does *not* track how many stops a visitor has personally clicked through, and does not change depending on which fork brought them there ‚Äî a branching tour has no single path length to count against, and that complexity wasn't asked for. Don't build a "path-aware" remaining-steps counter without checking first.

**Linking between two sibling tours is real navigation, not an overlay.** Where one tour's chain ends, a fork can point to a different tour file entirely (e.g. `tour_prometheus_eras.html`'s last stop links to `tour_serpent_forms.html`). Render this as a plain `<a href>` styled like the other fork buttons, not a JS redirect, so right-click / open-in-new-tab still works. This is a different kind of link than Deep Dive or Learn More ‚Äî it's deliberately leaving the current tour for a related one, not staying inside an overlay on the same page.

## Known gotchas

**Relative paths only.** The site is served from a GitHub Pages project path (`/kingdomofthecenter/`), not the domain root. Root-absolute paths (`/images/...`) will 404 in production. All `src` and `href` values must be relative.

**Inline script validation.** A stray duplicate line or broken brace in an inline `<script>` takes the whole page down silently. After any edit to an inline script, validate with:
```bash
node --check <extracted-script>.js
```

**One change at a time.** Use exact-anchor find/replace, confirm the result, then proceed. This is especially important in the timeline and notes pages where the inline scripts are long.

**Mobile scroll ‚Äî `openMobileDetail`.** Three constraints must hold simultaneously on the mobile layout:
1. Do not set `document.body.style.overflow = 'hidden'` ‚Äî it freezes page scroll and traps the user on the map.
2. Do not use `scrollIntoView({ behavior: 'smooth' })` on the panel ‚Äî the animation traps touch-scroll on iOS.
3. After `detail.scrollTop = 0`, jump to the panel with a ~60ms deferred instant `scrollIntoView({ block: 'start' })` ‚Äî this wins the scroll race against the selected row's own scroll.

**`escapeHTML` before any `innerHTML` injection.** `site-utils.js` exports `escapeHTML()` (replaces `&` first, then `<`, `>`, `"`, `'`). Route all sheet or CSV values through it before they reach `innerHTML`. This is already applied throughout; maintain it in any new injection points.

**`story_url` in sites data.** The "Read the story ‚Üí" button in a site's panel loads a URL into an iframe via `openStory()`. This value comes from the `story_url` column in the Google Sheet (or local `sites.csv`). It must be a **relative path** ‚Äî e.g. `tour_birds-without-borders.html` for a tour in the same `portabletemple/` folder. Never use an absolute domain URL here; it will break whenever the site is moved or the repo is renamed.

## Open investigations (not yet resolved)

**Learn More wiring is less convenient than Deep Dive / Read the Story, and could use a better approach.** `deepdive_url` and `story_url` are Google Sheet/CSV columns ‚Äî fill one in and the button appears, plain and visible at a glance. "Learn More" works differently: a site's button only appears if its `id` happens to match a key already present in `learn-more.json`, and that match-and-render logic lives in code (`timeline-pt.html`'s `LEARN_MORE_STEPS` check), not in a Sheet column. There's no single place to see "does this site have a Learn More," and a typo'd id fails silently with no error. Worth investigating a cleaner approach someday ‚Äî e.g. a `learn_more_id` column in the Sheet itself, or some validation step that flags ids in `learn-more.json` with no matching site. Not urgent; flagging so it isn't lost.

**Established behaviour: supplementary content opens as an overlay, never a new tab/window.** This applies to all three existing mechanisms ‚Äî `deepdive_url` (fetches a Google Doc's published HTML, strips Google's editor chrome, renders it in the `#deepdive-overlay` slide-in panel), `story_url` (loads the target page in an iframe, also inside a slide-in panel), and `learn-more.json` (renders an image+caption gallery in its own slide-in panel). None of them navigate away from the page the visitor is on. **Mistake made once, now corrected:** an early pass on a standalone tour file (`tour_prometheus_v2.html`) wired its Deep Dive button to `window.open(url, '_blank')`, which opens a new tab ‚Äî inconsistent with the rest of the site, and confusing precisely because it looks like a bug rather than a different design. See "Standalone tours" above for the full set of patterns worked out fixing this (overlay mechanics, hub-and-spoke navigation, button labeling) ‚Äî that section is the one to follow on the next tour, this one is just the history of how the mistake happened. Whether the three mechanisms should eventually be unified into one is a separate, larger question ‚Äî deliberately not addressed here; the goal right now is a reproducible, consistent prototype, not a redesign.
